# 多模态平台待澄清问题

生成时间：2026-04-22

以下问题会直接影响数据库设计、ETL 规则和平台接口，开发前需要尽快定口径。

## 1. 平台内部 user_id 的生成方式

虽然来源侧用户键已经定为 `标准姓名 + 手机号`，但平台内部 `user_id` 仍需要明确：

- 是否采用 UUID
- 是否允许人工合并/拆分用户
- 是否保留历史版本

## 2. 同日双设备记录如何合并

同一用户、同一天、同一时段，中科和玉生堂各采一次时：

- 算一个 visit 的双设备记录
- 还是两个平行 visit

这会直接影响 `fact_visit` 和 `mart_user_day_panel`。

## 3. visit_slot 的业务定义

需要确认：

- `早/中/晚` 是固定时间窗
- 还是当天第 1/2/3 次有效采集

如果后续研究主要做依从性分析，这个定义必须稳定。

## 4. 完整 visit 的来源级标准

需要分别定义：

- 中科一次完整 visit 必须具备哪些模态
- 玉生堂一次完整 visit 必须具备哪些模态

当前不能强行用同一个全局标准。

## 5. 质量规则的版本管理

需要明确：

- 作弊规则是否允许多版本并存
- 历史统计是否按规则版本冻结
- 数据重算是否覆盖旧结果

建议答案是：允许版本化，不覆盖历史快照。

## 6. 中西医扩展边界

平台未来如果接入：

- 检验
- 影像
- 生命体征
- 用药
- 诊断编码
- 可穿戴设备

需要提前确定统一编码体系和时间轴模型。

## 7. 脱敏与权限

需要明确：

- 哪些角色能看姓名和手机号
- 研究分析是否默认只看脱敏 ID
- 是否允许导出原始文件路径和 PDF

这会影响 API 和数据库字段暴露策略。

## 8. 原始资产存储策略

需要明确：

- 文件本体是否迁移到统一对象存储
- 数据库是否只存 URI 和 hash
- 原始资产是否允许覆盖

建议数据库只存元数据，不存大文件本体。

## 9. 首期平台交付边界

开发前最好明确第一阶段范围：

1. 只做数据库和 ETL
2. 加查询 API
3. 加研究分析面板
4. 加病例级多模态浏览

如果不先定边界，开发范围会持续膨胀。
