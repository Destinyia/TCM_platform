# 多模态数据类型存储索引展示设计

生成时间：2026-05-16

本文基于 `docs/多模态数据平台数据结构设计.md` 和当前冻结规则 `cohort_rule_v1_20260422`，用于后续 ETL、数据库、API 和前端病例浏览开发。

## 1. 当前入库规则摘要

### 1.1 身份与 visit 规则

- 平台内部用户主键使用独立 `user_id`，不直接等于任何来源字段。
- 来源侧用户键统一为 `source_user_key = canonical_name + phone`。
- 中科来源侧关键凭据为 `病例号`，但同一病例号可能对应多个模态采集文件和相近采集时间；标准入库需同时保存 `source_visit_id=病例号`、`source_record_group_id`、各模态 `collected_at`。
- 玉生堂 visit 来源键为 `TreatNumber`。
- 玉生堂 `CaId` 只作为辅助来源 ID 保留，不作为平台用户主键。
- 原始姓名、标准姓名、手机号、来源路径、来源字段必须长期保留。
- 文件夹姓名只能作为审计线索，不能作为身份主依据；中科以表格内姓名/病例号/采集时间为准，玉生堂以结构化 JSON、PDF 报告和手机号/CaId 映射为准。
- 玉生堂身份映射优先级为 `CaId` > `PDF姓名` > `PDF脱敏电话`；PDF 仅作兜底补充，不覆盖结构化字段。

### 1.2 来源记录、visit 与打卡矩阵规则

- 研究主线是用户纵向随访。
- 标准入库层必须保留原始记录粒度，不删除、不覆盖：一个玉生堂原始病例目录一条原始记录；中科按病例号、姓名、采集时间和模态文件保留来源证据。
- 中科同一 `病例号 + 标准姓名` 下，采集时间相近且在 5 分钟以内的不同模态文件，视为同一次来源采集记录，合并模态存在性、路径和缺失信息；备注或质量证据记录参与合并的具体时间。
- 同一用户、同一天内，间隔 30 分钟以内的记录视为同一候选研究时段；时间重叠时，每个设备保留模态最完整的一条作为有效代表。
- 研究型 `visit_slot` 可以按当天第 1/2/3 次有效采集得到早/中/晚；当前 4 月打卡矩阵和审计表使用固定时间窗：`早 < 11:00`，`中 11:00-15:59:59`，`晚 >= 16:00`。
- 同日同槽位的中科和玉生堂记录在 `mart_user_day_panel` 中合并为一个研究视角下的用户时段，但来源侧仍应保留各自 `fact_visit`、`fact_modality_record`、`fact_file_asset`。
- 时间重叠时，保留模态最完整的一次作为有效代表时间。
- 超出早/中/晚三次的记录在审计矩阵中显示实际次数，不能简单截断为 1；研究层可按规则标记 `overflow_gt_3` 或作为额外记录单独审计。
- 重复导出的玉生堂病例目录不参与矩阵有效打卡计数，统一排列在详情表末尾，并在 `visit_slot/时段` 标注 `重复`。
- 当前审计导出需区分 `玉生堂打卡详情` 与 `中科打卡详情` 两个 sheet；矩阵合并统计次数，另设个人汇总 sheet 统计每人有效打卡次数/全部打卡次数。

### 1.3 质量规则

标准质量标签：

- `missing_modality`
- `parse_failed`
- `duplicate_export`
- `duplicate_numeric_exact`
- `duplicate_numeric_similar`
- `triplicate_within_10m`
- `gap_lt_threshold`
- `name_alias_mapped`
- `name_mismatch`
- `pdf_identity_fallback`
- `report_only_record`
- `source_caseid_time_cluster_5m`
- `overflow_gt_3`

标准质量状态：

- `valid`
- `incomplete`
- `suspicious`
- `invalid`
- `duplicate`

质量规则必须版本化，至少记录 `rule_version` 和 `pipeline_version`。

## 2. 平台统一模态字典

| 模态编码 | 中文名称 | 数据性质 | 当前来源 | 是否核心模态 | 说明 |
| --- | --- | --- | --- | --- | --- |

| `pulse` | 脉诊 | 结构化、波形、图片 | 中科、玉生堂 | 是 | 包括脉象结论、数值特征、左右手或部位波形 |

| `face` | 面诊 | 结构化、图片 | 中科为主 | 是 | 包括面色、唇色、光泽、区域指标、面图 |
| `voice` | 声诊/语音 | 结构化、音频 | 玉生堂 | 扩展核心 | 包括声学特征、zip、wav |
| `western_exam` | 西医/体检扩展 | 结构化 | 玉生堂 | 扩展 | 当前多为空数组，但需保留扩展位 |

| `order` | 订单/挂号/签名 | 结构化、图片 | 玉生堂 | 管理辅助 | 支付、排号、签名图 |
| `basic` | 基础体征 | 结构化 | 玉生堂、中科字段 | 辅助 | 身高、体重、血压、体温、血糖等 |

## 3. 来源数据到模态映射

### 3.1 中科四诊仪

| 数据类别 | 来源文件/字段 | 标准模态 | 存储对象 | 备注 |
| --- | --- | --- | --- | --- |
| 问诊表 | 问诊 `.xls`、病例详情体质字段 | `ask` | `fact_modality_record.parsed_structured_data_json` | `.xls` 可能是真 OLE/BIFF，需按 magic bytes 选解析器 |
| 脉诊表 | 脉诊 `.xls/.csv` | `pulse` | 结构化 JSON + 特征摘要 | 保存脉率、节律、脉力、h1-h5、t/t1/t4/t5、W、As、Ad 等 |
| 脉波原始文件 | 无扩展名 `left/right` | `pulse` | `fact_file_asset` 或专门 raw signal 表 | 不按扩展名识别，按二进制资产保留 |
| 舌诊表 | 舌诊 `.xls/.csv` | `tongue` | 结构化 JSON + 特征摘要 | 保存舌色、苔色、胖瘦、厚薄、腐腻、裂纹、Lab/RGB 指标 |
| 舌图 | 原图、分割、分区 JPG | `tongue` | `fact_file_asset` | `asset_type` 区分 `tongue_origin`、`tongue_segmented`、`tongue_region` |
| 面诊表 | 面诊 `.xls/.csv` | `face` | 结构化 JSON + 特征摘要 | 保存面色、唇色、光泽、上/中/下庭和区域颜色指标 |
| 面图 | 原图、分区 JPG | `face` | `fact_file_asset` | 一代目录中 `m_*.jpg`、`s_*.jpg` 需通过规则确认角色 |
| PDF 报告 | `1.0报告.pdf`、`2.0报告.pdf` | `report` | `fact_file_asset` | `asset_type=report_pdf` |
| 穴位/建议图片 | `nei_guan.jpg`、`qie_men.jpg`、`.png` 素材 | `prescription` 或 `report` 辅助 | `fact_file_asset` | 不能混入舌图/面图训练数据 |

**中科解析与聚合补充规则**

- 身份识别不依赖目录名；目录名可能是错误、别名或伪造信息，只作为 `folder_name` 审计字段。
- 优先使用表格内姓名，经 `config/name_alias_rules_v1.json` 映射为 `canonical_name`；原始表格姓名写入 `raw_name/table_name`。
- `病例号` 写入 `source_visit_id`，并作为详情表第二关键列展示。
- 同一 `病例号 + canonical_name` 下，多个模态文件的 `collected_at` 相差 5 分钟以内时，合并为同一来源采集记录；`source_record_group_id` 可由 `source_vendor + source_visit_id + canonical_name + cluster_start` 生成。
- 合并后的采集时间取该组最早 `collected_at`，并在 `source_evidence_json` 记录组内所有原始时间和文件路径。
- 中科完整性当前按问诊、脉诊、舌诊、面诊四类判断；缺少哪一类必须明确写入 `missing_modalities`，不能只写“不完整”。

### 3.2 玉生堂四诊仪

| 数据类别 | 来源文件/字段 | 标准模态 | 存储对象 | 备注 |
| --- | --- | --- | --- | --- |
| 客户档案 | `customerArchive.json` | `basic` | 用户身份映射 + 基础信息 JSON | 保留 `CaId`、姓名、手机号、生日、地址等 |
| 问诊 | `dataAsk.json` | `ask` | 结构化 JSON | 常为问题数组，保留题目 ID 与答案 |
| 基础指标 | `dataBasic.json` | `basic` | 结构化 JSON | 身高、体重、体温、血压、脉率、声学基础指标 |
| 诊疗记录 | `treatRecord.json` | `basic` 或 visit 元数据 | `fact_visit` 扩展 JSON | 诊疗时间、主诉、结论、医嘱 |
| 脉诊 | `pulse/dataPulse.json`、`dataPulseRight.json` | `pulse` | 结构化 JSON + raw signal | 大段波形字符串建议拆到 raw signal 或对象存储 |
| 脉图 | `pulse_0.jpg`、`pulse_1.jpg`、`pulseImage.jpg` | `pulse` | `fact_file_asset` | `asset_type=pulse_image` |
| 舌诊 | `tongue/dataTongue.json` | `tongue` | 结构化 JSON | 舌图路径、舌色、厚薄、裂纹、专家分析 |
| 舌图 | `OriginPic.jpg`、`CutPic.jpg`、`RectanglePic.jpg` | `tongue` | `fact_file_asset` | 原图、裁剪图、框选/区域图分别建资产 |
| 声诊 | `voice/dataVoice.json` | `voice` | 结构化 JSON + 特征摘要 | `AnalysisResults` 可能是嵌套 JSON 字符串 |
| 语音文件 | `voice/*.zip`、`*.wav` | `voice` | `fact_file_asset` | zip 与 wav 分别登记资产，wav 可派生声学特征 |
| 西医扩展 | `western/dataWestern.json` | `western_exam` | 结构化 JSON | 当前多为空数组，但保留扩展能力 |
| 处方 | `treatPrescription.json` | `prescription` | 结构化 JSON | `Detail` 可能是嵌套 JSON 字符串 |
| 订单/签名 | `treatOrder.json`、`numberOrder.json`、`SignPicture.jpg` | `order` | 结构化 JSON + 文件资产 | 管理辅助，不默认进研究特征 |
| PDF 报告 | `{TreatNumber}.pdf` | `report` | `fact_file_asset` | `asset_type=report_pdf` |

**玉生堂解析与聚合补充规则**

- 一个原始病例目录对应一条原始记录，`TreatNumber/病例目录名` 写入 `source_visit_id`，并作为详情表第二关键列展示。
- 时间字段优先级：结构化记录内部时间 > PDF 报告评估时间 > 病例号前缀时间。
- 身份映射优先级：`CaId` > PDF 解析姓名 > PDF 脱敏电话；`CaId`、PDF 姓名、PDF 电话均需作为证据字段保留。
- 当 `dataAsk.json` 为空数组但 PDF 可解析出姓名和评估时间时，记录仍应入库，质量状态为 `incomplete` 或 `report_only_record`，不得丢弃。
- 重复导出的相同病例目录或相同 `TreatNumber` 不参与矩阵有效计数，但要保留审计行，标记 `duplicate_export`。

## 4. 各模态存储、索引与展示设计

### 4.1 问诊/问卷/体质 `ask`

**包含数据**

- 中科问诊汇总、题库 sheet、体质辨识、五态人格。
- 玉生堂 `dataAsk.json`、主诉、基础问诊答案。

**存储方式**

- 原始 Excel/JSON 进入 `raw` 层并登记 `fact_file_asset`。
- 标准层在 `fact_modality_record` 写入：
  - `modality_type=ask`
  - `parsed_structured_data_json`：题目、答案、题目 ID、来源字段名、体质结论。
  - `feature_summary_json`：九种体质分值、主导体质、问卷完成度。
- 后续可拆分为：
  - `feature_questionnaire_response`
  - `feature_constitution_score`

**索引方式**

- 主索引：`visit_id`、`user_id`、`visit_date`。
- 来源索引：`source_vendor`、`source_visit_id`。
- JSON 索引：体质类型、题目 ID、主诉关键词。
- 质量索引：`completion_status`、`quality_flags_json`。

**展示形式**

- 病例详情页中的问诊卡片：主诉、体质结论、关键异常答案。
- 问卷明细表：题目、答案、选项、来源。
- 体质雷达图或柱状图：九种体质分值。
- 纵向趋势：同一用户体质分值随访曲线。

### 4.2 脉诊 `pulse`

**包含数据**

- 中科脉诊表、左右手无扩展名原始波形文件。
- 玉生堂 `dataPulse.json`、`dataPulseRight.json`、脉图 JPG、部位波形字符串。

**存储方式**

- 结构化结论写入 `fact_modality_record.parsed_structured_data_json`。
- 高频或大段波形不直接展开到宽表，建议采用两层：
  - `fact_file_asset`：登记原始 binary、zip、图片资产。
  - `feature_pulse_raw_signal` 或对象存储：保存波形数组、采样率、侧别、部位。
- `feature_summary_json` 保存常用统计特征，例如脉率、节律、强度、速度、幅度、h1-h5、W、As、Ad。
- `numeric_fingerprint` 用于数值重复、相似重复检测。

**索引方式**

- 主索引：`visit_id`、`modality_type=pulse`。
- 侧别/部位索引：`side=left/right`、`position=cun/guan/chi`。
- 数值查重索引：`numeric_fingerprint`。
- 资产索引：`asset_type=pulse_raw_left/right`、`pulse_image`。
- 研究索引：脉象类型、脉率范围、异常标记。

**展示形式**

- 左右手/寸关尺波形图。
- 脉率、节律、脉力、紧张度等指标面板。
- 脉象标签列表和结构化结论。
- 同一用户纵向脉率、脉力、重复采集异常趋势。
- 原始波形资产下载或预览入口。

### 4.3 舌诊 `tongue`

**包含数据**

- 中科舌诊表、原图、分割图、分区图。
- 玉生堂 `dataTongue.json`、`OriginPic.jpg`、`CutPic.jpg`、`RectanglePic.jpg`。

**存储方式**

- 舌诊结构化字段写入 `parsed_structured_data_json`。
- 舌图进入 `fact_file_asset`：
  - `asset_type=tongue_origin`
  - `asset_type=tongue_segmented`
  - `asset_type=tongue_region`
  - `asset_type=tongue_cut`
- 图像派生特征写入 `feature_summary_json`，例如舌色、苔色、厚薄、胖瘦、裂纹、腐腻、Lab/RGB 指标。
- 训练数据使用 `mart` 或快照表输出，不直接从目录扫图。

**索引方式**

- 主索引：`visit_id`、`modality_type=tongue`。
- 资产索引：`file_hash`、`asset_type`、`storage_uri`。
- 特征索引：舌色、苔色、厚薄、裂纹、胖瘦。
- 图像检索索引：可后续增加图像 embedding，按 `asset_id` 关联。

**展示形式**

- 舌图对照查看：原图、裁剪、分割、分区并排。
- 舌象结构化标签：舌色、苔色、厚薄、裂纹等。
- 颜色指标表或色块展示。
- 纵向舌图时间轴。
- 训练集审核视图：只展示诊断图像，排除穴位/建议素材图。

### 4.4 面诊 `face`

**包含数据**

- 中科面诊表、面部原图、分区图。
- 玉生堂当前缺少稳定独立面诊数据，暂不强行补齐。

**存储方式**

- 面诊结构化字段写入 `parsed_structured_data_json`。
- 面图资产写入 `fact_file_asset`：
  - `asset_type=face_origin`
  - `asset_type=face_region`
  - `asset_type=face_processed`
- `feature_summary_json` 保存面色、唇色、光泽、上/中/下庭、鼻、唇、眼周等区域指标。

**索引方式**

- 主索引：`visit_id`、`modality_type=face`。
- 特征索引：面色、唇色、光泽、区域异常。
- 资产索引：`asset_type`、`file_hash`。
- 缺失索引：玉生堂 visit 下 `face.exists_flag=false` 不作为异常，按来源完整性规则判断。

**展示形式**

- 面图原图/分区图切换。
- 面色、唇色、光泽和区域分析摘要。
- 区域指标表。
- 用户随访时间轴中的面图缩略图。

### 4.5 声诊/语音 `voice`

**包含数据**

- 玉生堂 `dataVoice.json`。
- `voice/*.zip` 与解压后的 `.wav`。
- 中科当前未见稳定声诊数据。

**存储方式**

- 声学结构化结果写入 `parsed_structured_data_json`。
- `AnalysisResults` 等嵌套 JSON 字符串需二次解析，失败时保留原始字符串并打 `parse_failed`。
- zip、wav 登记到 `fact_file_asset`：
  - `asset_type=voice_zip`
  - `asset_type=voice_wav`
- 派生声学特征写入 `feature_summary_json`，例如 Pitch、RMS、ZCR、能量、熵等。

**索引方式**

- 主索引：`visit_id`、`modality_type=voice`。
- 资产索引：`asset_type`、`file_hash`、`mime_type`。
- 特征索引：音节/字音、Pitch 范围、RMS、ZCR。
- 质量索引：缺失音频、zip 解析失败、wav 数量异常。

**展示形式**

- 音频播放器。
- 音频文件列表：zip、wav、音节名称。
- 波形图和声谱图。
- 声学特征表。
- 纵向语音特征趋势。

### 4.6 西医/体检扩展 `western_exam`

**包含数据**

- 玉生堂 `western/dataWestern.json`。
- 后续可能接入检验、影像、生命体征、诊断编码。

**存储方式**

- 当前为空数组也应生成 `fact_modality_record`，用于显式表达“存在但为空”或“缺失”。
- 结构化结果写入 `parsed_structured_data_json`。
- 后续高价值字段可拆分为专门特征表，例如检验指标表、诊断编码表。

**索引方式**

- 主索引：`visit_id`、`modality_type=western_exam`。
- 项目索引：检查项目编码、项目名称、异常标志。
- 时间索引：检查时间、报告时间。

**展示形式**

- 检查项目表。
- 异常指标高亮。
- 与中医模态同一时间轴展示。

### 4.7 PDF 报告 `report`

**包含数据**

- 中科 `1.0报告.pdf`、`2.0报告.pdf`。
- 玉生堂 `{TreatNumber}.pdf`。

**存储方式**

- PDF 只作为文件资产登记到 `fact_file_asset`。
- 不建议把 PDF 本体写入数据库。
- 可后续增加 OCR 或结构化抽取结果，作为派生 `report_text` 或 `report_extract_json`。

**索引方式**

- 主索引：`visit_id`。
- 资产索引：`asset_type=report_pdf`、`file_hash`。
- 来源索引：`source_vendor`、`source_visit_id`。

**展示形式**

- 病例详情页 PDF 预览。
- 下载入口。
- 报告版本标签：中科 1.0/2.0、玉生堂报告。

### 4.8 处方/调理建议 `prescription`

**包含数据**

- 玉生堂 `treatPrescription.json`。
- 中科病例详情中的饮食调养、运动调理、穴位/调理建议。
- 中科穴位或调理建议图片。

**存储方式**

- 结构化处方写入 `parsed_structured_data_json`。
- 药味明细、用法、疗程、总量应尽量二次解析。
- 建议类图片登记为 `fact_file_asset`，但 `asset_role` 必须标明 `recommendation_material`。

**索引方式**

- 主索引：`visit_id`、`modality_type=prescription`。
- 药物/建议索引：处方名、药味名、调理类型。
- 资产索引：穴位图、建议图。

**展示形式**

- 处方卡片：处方名、用法、疗程。
- 药味明细表。
- 饮食/运动/穴位建议分组展示。
- 建议图片预览。

### 4.9 订单/挂号/签名 `order`

**包含数据**

- 玉生堂 `treatOrder.json`、`numberOrder.json`、`SignPicture.jpg`。

**存储方式**

- 订单和挂号信息写入 `parsed_structured_data_json`。
- 签名图作为 `fact_file_asset`，`asset_type=signature_image`。
- 默认不进入研究特征层，只用于追溯和管理。

**索引方式**

- 主索引：`visit_id`、`source_visit_id`。
- 管理索引：订单号、支付状态、挂号时间。
- 资产索引：签名图 hash。

**展示形式**

- 管理信息折叠区。
- 签名图预览。
- 追溯信息表。

### 4.10 基础体征 `basic`

**包含数据**

- 身高、体重、体温、血压、血糖、脉率、生日、性别等。
- 来源包括中科表格字段、玉生堂 `dataBasic.json` 和汇总 Excel。

**存储方式**

- 用户稳定属性优先写入 `dim_user` 或身份映射表。
- visit 级体征写入 `fact_modality_record`，`modality_type=basic`。
- 可后续拆为 `feature_vital_sign`。

**索引方式**

- 用户索引：`user_id`、手机号、标准姓名。
- 时间索引：`visit_date`。
- 指标索引：血压、血糖、BMI、体温异常。

**展示形式**

- 病例基本信息栏。
- 生命体征指标卡。
- BMI、血压、血糖纵向趋势。

## 7. 前端展示页面建议

### 7.1 用户纵向随访页

核心数据源：`mart_user_day_panel`。

展示内容：

- 日历热力图：早/中/晚完成情况；当前审计矩阵按 `<11:00`、`11:00-15:59:59`、`>=16:00` 三个固定时间窗统计，单元格显示实际次数。
- 模态完整度矩阵：问诊、脉诊、舌诊、面诊、声诊、报告。
- 质量标记：缺失、疑似作弊、重复、姓名冲突、PDF 兜底、同病例号 5 分钟合并。
- 纵向趋势入口：体质、脉率、血压、舌象标签。
- 个人汇总：每人有效打卡次数、全部打卡次数，并按中科/玉生堂拆分。

### 7.2 visit 多模态详情页

核心数据源：`fact_visit`、`fact_modality_record`、`fact_file_asset`。

展示内容：

- visit 元数据：来源、来源 visit ID、采集时间、时段、质量状态。
- 关键凭据：中科显示病例号；玉生堂显示 TreatNumber/病例目录名；如由 PDF 兜底解析，显示 PDF 姓名和 PDF 评估时间。
- 模态标签页：问诊、脉诊、舌诊、面诊、声诊、西医、处方、报告。
- 每个模态显示结构化摘要、文件资产、解析状态。
- 中科同病例号 5 分钟内多模态合并时，展示组内原始采集时间列表和参与合并的文件路径。
- 原始路径和 hash 只对有权限角色展示。

### 7.3 资产审核页

核心数据源：`fact_file_asset`。

展示内容：

- 按资产类型筛选：舌图、面图、脉波、语音、PDF、建议图片。
- 展示 hash、大小、解析状态、训练可用标记。
- 支持将建议图片、穴位图片标记为 `is_training_eligible=false`。

### 7.4 数据质量页

核心数据源：`fact_visit`、`fact_modality_record`、`mart_user_day_panel`。

展示内容：

- 缺失模态统计。
- 重复导出、同病例号短时间多模态合并、数值重复/高度相似列表。
- 10 分钟内多次采集列表。
- 姓名别名映射和姓名冲突列表。
- PDF 兜底解析记录、仅报告记录、结构化 JSON 为空数组记录。
- 按规则版本筛选。

### 7.5 研究数据集快照页

核心数据源：`mart` 层快照。

展示内容：

- cohort、日期范围、模态组合筛选。
- 纳入/排除条件。
- 样本量、用户数、visit 数、模态覆盖率。
- 导出任务和数据版本。

## 8. ETL 开发约束

1. 不按扩展名直接选择解析器，必须读取 magic bytes。
2. Excel/CSV 中的病例号、TreatNumber、CaId、手机号必须按字符串处理。
3. CSV 编码按 `utf-8-sig -> utf-8 -> gb18030` 顺序尝试，并用字段字典校验。
4. 嵌套 JSON 字符串必须二次解析，失败时保留原文并打 `parse_failed`。
5. 文件资产一律登记原始路径、统一 URI、hash、大小、mime type。
6. 同一来源重复导出不覆盖原始记录，使用来源键和 hash 去重，并保留 `duplicate_export` 标记。
7. 来源完整性规则按 vendor 区分，不用一个全局“完整 visit”标准判断所有来源。
8. 训练集只从标准资产表和快照表生成，不直接扫原始目录。
9. 中科同病例号 5 分钟内多模态合并只影响标准来源采集记录和审计矩阵计数，不删除原始模态文件资产。
10. 当前打卡审计导出必须包含矩阵、玉生堂详情、中科详情、个人汇总和规则说明；详情行中的路径列应提供可回溯到原始文件的链接。
11. 文件夹姓名与表格/PDF/JSON 姓名不一致时仍按可信来源字段统计，但必须写入 `name_mismatch` 和备注。
