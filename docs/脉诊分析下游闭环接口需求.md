# 通用研究数据下载与脉诊模块低耦合边界

更新时间：2026-05-20

## 1. 定位

脉诊分析模块是数据平台的一个独立下游消费者，不应与数据平台强耦合。数据平台不应为了脉诊模块暴露一组只能服务脉诊的专用下载契约；它应提供通用、版本化、脱敏、可审计的数据集下载能力，供脉诊、舌诊、面诊、声诊、多模态融合等独立研究者复用。

脉诊模块可以作为第一批消费者来验证平台下载能力是否足够，但验证目标是反推通用能力，而不是把平台接口固化成脉诊模块内部数据结构。

推荐边界为：

```text
原始文件入库
  -> 标准化 visit / modality / asset
  -> 通用结构化 records / 宽表 / 高维明细 / 资产索引
  -> 通用数据集版本构建
  -> manifest / Parquet / CSV / JSONL / 文件包下载
  -> 独立研究者在外部模块中消费
  -> 可选：研究结果以独立结果集方式导入平台
```

平台核心职责：数据治理、脱敏、索引、版本、下载、审计。

下游模块职责：领域算法、质量建模、统计分析、报告生成、实验复现。

## 2. 当前已满足的输入

当前 M2 已具备下游分析的最小雏形：

| 能力 | 当前状态 |
| --- | --- |
| 标准 visit | 已有 `fact_visit` |
| 模态记录 | 已有 `fact_modality_record` |
| 文件资产索引 | 已有 `fact_file_asset` |
| 结构化脉诊 records | 已写入 `parsed_structured_data_json.records[]` |
| visit 级宽表 | 已有 `mart_visit_feature_wide`，当前版本 `visit_feature_wide_v2` |
| 脉诊 records 查询 | 已有 `GET /api/demo/pulse/records` |
| 基础趋势和聚合展示 | 已有 `/pulse-analysis` 页面 |

这些能力足够支撑展示型分析，但还不足以作为通用研究下载能力交付给独立研究者。

2026-05-20 一阶段开发已开始，当前已新增 `dim_device`、`dim_feature_variable`、`fact_pulse_measurement`、`fact_pulse_waveform_asset`、`fact_pulse_position_feature`、`analysis_run`、`fact_pulse_measurement_quality`、`fact_pulse_feature_reliability`，并提供 `/api/pulse/*` 查询接口。更正后，这些内容定位为脉诊研究模块的本地验证结构和辅助 API，不作为数据平台面向其他研究者的通用下载契约。

## 3. 平台通用下载能力

### 3.1 通用数据集版本

平台应围绕 `dataset_definition`、`dataset_version`、`dataset_item` 或等价结构组织下载，而不是围绕某个下游算法模块组织下载。

必须字段：

| 字段 | 说明 |
| --- | --- |
| `dataset_id` | 数据集定义 ID |
| `version_id` | 不可变版本 ID |
| `version_name` | 版本名 |
| `task_type` | 任务类型，可为 `pulse_quality`、`tongue_color`、`multimodal` 等 |
| `modality_filter_json` | 模态过滤 |
| `quality_filter_json` | 质量策略 |
| `asset_policy_json` | 文件资产导出策略 |
| `deidentify_policy_json` | 脱敏策略 |
| `manifest_uri` | manifest 地址 |
| `artifact_uri` | 下载包地址 |
| `summary_json` | 样本量、用户数、模态覆盖等摘要 |

### 3.2 通用 manifest

必须字段：

| 字段 | 说明 |
| --- | --- |
| `sample_id` | 数据集内样本 ID |
| `user_id` | 脱敏用户 ID |
| `visit_id` | 平台 visit ID |
| `source_vendor` | 来源平台 |
| `visit_time` / `visit_date` / `visit_slot` | 时间信息 |
| `modalities` | 样本包含模态 |
| `quality_status` / `quality_flags` | 质量字段 |
| `features_uri` | 该样本结构化特征位置 |
| `assets` | 文件资产列表 |
| `labels` | 标签信息，如有 |
| `provenance` | 来源追溯信息，脱敏后提供 |

### 3.3 通用下载包

推荐结构：

```text
datasets/{dataset_id}/{version}/
  dataset_card.md
  manifest.jsonl
  visits.parquet
  modality_records.parquet
  feature_wide.parquet
  feature_variables.parquet
  high_dim/
    pulse_position_features.parquet
    waveform_manifest.jsonl
    tongue_region_color.parquet
    voice_item_features.parquet
  assets_manifest.jsonl
  files/
```

### 3.4 通用变量字典

`dim_feature_variable` 应服务所有模态，不应只服务脉诊。

必须字段：

| 字段 | 说明 |
| --- | --- |
| `feature_name` | 标准变量名 |
| `display_name` | 中文名 |
| `modality_type` | 模态 |
| `feature_level` | visit/modality/region/segment/window/beat 等 |
| `source_vendor` | 来源，或 `standard` |
| `data_type` | numeric/categorical/text/json |
| `unit` | 单位 |
| `category` | 质量/颜色/时域/频域/形态/声学/问诊等 |
| `is_ml_feature` | 是否可进入机器学习样本 |
| `is_quality_feature` | 是否为质量控制字段 |
| `valid_range_json` | 合理范围 |
| `description` | 说明 |

## 4. 平台通用下载 API

### 4.1 数据集构建

用于将平台数据固化为独立研究者可复现消费的版本。

```text
POST /api/datasets
POST /api/datasets/{dataset_id}/versions
GET  /api/datasets/{dataset_id}/versions/{version_id}
GET  /api/datasets/{dataset_id}/versions/{version_id}/manifest
GET  /api/datasets/{dataset_id}/versions/{version_id}/artifact
```

通用过滤：

| 过滤项 | 说明 |
| --- | --- |
| `modalities` | 一个或多个模态 |
| `source_vendor` | 中科/玉生堂 |
| `date_range` | 日期范围 |
| `user_ids` | 用户集合 |
| `visit_slots` | 早/中/晚 |
| `quality_status` | visit 质量 |
| `asset_policy` | 是否包含原始资产、预览资产、标准化资产 |
| `include_feature_wide` | 是否包含宽表 |
| `include_high_dim` | 是否包含高维明细 |
| `deidentify` | 是否脱敏，研究导出默认 true |

### 4.2 下载接口

导出接口不应要求研究者了解平台内部表结构。

```text
GET /api/datasets/{dataset_id}/versions/{version_id}/manifest
GET /api/datasets/{dataset_id}/versions/{version_id}/artifact?format=zip
GET /api/datasets/{dataset_id}/versions/{version_id}/artifact?format=parquet
GET /api/datasets/{dataset_id}/versions/{version_id}/files/{file_id}
```

## 5. 脉诊模块低耦合消费方式

脉诊模块应通过以下方式消费平台数据：

```text
1. 选择或创建一个通用数据集版本，例如 modalities=["pulse"]。
2. 下载 manifest、feature_wide、高维明细和资产包。
3. 在脉诊模块内部构建 measurement/window/beat/feature 等分析结构。
4. 在脉诊模块内部生成质量报告、可靠片段和特征可靠性等级。
5. 如需要回到平台展示，作为独立 research_result 或 analysis_artifact 导入，而不是直接修改平台原始表。
```

当前 `/api/pulse/*` 接口可保留为本仓库内的脉诊模块辅助 API，但不作为未来独立研究者下载数据的主要入口。

## 6. 研究结果导入（可选）

平台可以支持研究结果导入，但应保持结果集与原始治理数据分离。

建议结构：

| 表 | 用途 |
| --- | --- |
| `analysis_run` | 一次分析运行，记录代码版本、参数、输入数据集版本 |
| `analysis_artifact` | 报告、图表、指标文件、结果 Parquet |
| `analysis_result_item` | 可选的样本级结果索引 |

可选接口：

```text
POST /api/analysis-runs
POST /api/analysis-runs/{run_id}/artifacts
POST /api/analysis-runs/{run_id}/result-items
GET  /api/analysis-runs/{run_id}
```

## 7. MVP 优先级

为了同时支持脉诊模块和其他独立研究者，M2 剩余工作应优先转向通用下载能力：

1. 增加通用 `dataset_definition` 和完善 `dataset_version`，补 `manifest_uri`、`artifact_uri`、`deidentify_policy_json`、`asset_policy_json`。
2. 增加通用 dataset build 服务，可从 visit、modality records、宽表、高维明细和资产索引生成版本快照。
3. 增加 `manifest.jsonl` 导出。
4. 增加 `feature_wide.parquet`、`modality_records.parquet`、`assets_manifest.jsonl` 导出。
5. 支持按模态选择高维明细，例如 pulse waveform manifest、tongue region color、voice item features。
6. 支持脱敏下载和导出审计。
7. 脉诊模块仅作为第一个下载消费者，用下载包构建自身分析结构。

完成以上 7 项后，数据平台可以服务多个独立研究方向；脉诊分析模块也可以在低耦合条件下复现实验。

当前已完成脉诊模块本地验证结构和辅助查询接口；通用 dataset manifest、CSV/Parquet/ZIP 下载和导出审计尚未完成。
