# 脉诊分析下游闭环接口需求

更新时间：2026-05-20

## 1. 定位

脉诊分析模块是数据平台的下游任务，不直接读取原始目录、Excel 或 JSON 文件。它只消费数据平台产出的标准数据集、manifest、结构化特征表、高维明细表和文件资产索引。

因此，脉诊模块的开发应反向约束数据平台必须提供的能力：数据平台不仅要能展示 visit 和 records，还要能稳定产出可复现、可追溯、可过滤、可导出的脉诊分析数据集。

闭环链路为：

```text
原始文件入库
  -> 标准化 visit / modality / asset
  -> 脉诊结构化解析
  -> 波形与六部位高维明细索引
  -> 变量字典与质量字段
  -> 数据集版本构建
  -> manifest / CSV / Parquet / 文件资产导出
  -> 脉诊可信度分析
  -> 质量结果、可靠片段、特征等级回写数据平台
```

## 2. 当前已满足的输入

当前 M2 已具备下游分析的最小雏形：

| 能力 | 当前状态 |
| --- | --- |
| 标准 visit | 已有 `fact_visit` |
| 模态记录 | 已有 `fact_modality_record` |
| 文件资产索引 | 已有 `fact_file_asset` |
| 结构化脉诊 records | 已写入 `parsed_structured_data_json.records[]` |
| visit 级宽表 | 已有 `mart_visit_feature_wide`，当前版本 `visit_feature_wide_v2` |
| 脉诊 records 查询 | 已有 `GET /api/demo/pulse/records` |
| 基础趋势和聚合展示 | 已有 `/pulse-analysis` 页面 |

这些能力足够支撑展示型分析，但不足以支撑 `docs/脉诊分析模块设计.md` 中的第一版可信度闭环。

2026-05-20 一阶段开发已开始，平台侧已新增 `dim_device`、`dim_feature_variable`、`fact_pulse_measurement`、`fact_pulse_waveform_asset`、`fact_pulse_position_feature`、`analysis_run`、`fact_pulse_measurement_quality`、`fact_pulse_feature_reliability`，并提供 `/api/pulse/*` analysis-ready 查询接口。当前 demo seed 可生成示例 measurement、变量字典、波形索引和部位特征；真实解析流程接入仍是后续重点。

## 3. 下游反推的平台必备数据结构

### 3.1 设备维表

已新增 `dim_device`，用于设备一致性分析。

必须字段：

| 字段 | 说明 |
| --- | --- |
| `device_id` | 平台内设备 ID |
| `source_vendor` | 来源平台 |
| `source_device_id` | 来源系统设备 ID，如能取得 |
| `device_model` | 设备型号 |
| `sensor_type` | 传感器类型 |
| `firmware_version` | 固件版本 |
| `sampling_rate` | 默认采样率 |
| `calibration_date` | 最近校准日期 |
| `device_meta_json` | 其他设备元数据 |

`fact_visit` 或脉诊 measurement 级记录必须能关联 `device_id`。如果原始数据暂时没有设备号，也应显式写入 `unknown`，不能缺省。

### 3.2 脉诊 measurement 表

已新增 `fact_pulse_measurement`，表达一次完整脉诊采集。它是下游分析的样本主表。

必须字段：

| 字段 | 说明 |
| --- | --- |
| `measurement_id` | 脉诊测量 ID |
| `visit_id` | 关联标准 visit |
| `modality_record_id` | 关联原始脉诊模态记录 |
| `user_id` | 用户 ID |
| `device_id` | 设备 ID |
| `source_vendor` | 来源平台 |
| `source_measurement_id` | 来源测量编号 |
| `start_time` / `end_time` | 采集起止时间 |
| `duration_seconds` | 采集时长 |
| `visit_slot` | 早/中/晚 |
| `collection_hour` | 连续小时变量 |
| `hand_side` | 左/右手 |
| `pulse_position` | 寸/关/尺/整体/未知 |
| `sampling_rate` | 实际采样率 |
| `quality_status` | 平台原始质量状态 |
| `source_meta_json` | 来源字段追溯 |

### 3.3 波形资产索引

已新增 `fact_pulse_waveform_asset`。完整波形不进入 visit 宽表。

必须字段：

| 字段 | 说明 |
| --- | --- |
| `waveform_asset_id` | 波形资产 ID |
| `measurement_id` | 所属 measurement |
| `asset_id` | 如落在 `fact_file_asset`，关联资产 ID |
| `channel_name` | `SinglePulse`、`Cun`、`GuanMai`、`Chi` 等 |
| `hand_side` | 左/右手 |
| `pulse_position` | 部位 |
| `sample_count` | 样本点数 |
| `sampling_rate` | 采样率 |
| `storage_uri` | Parquet/NPY/NPZ/bin 路径 |
| `data_format` | 文件格式 |
| `hash` | 内容 hash |
| `preview_json` | 前端预览降采样点 |
| `summary_json` | min/max/mean/std 等摘要 |

### 3.4 六部位明细表

已新增 `fact_pulse_position_feature`，承接中科六部位测量和后续部位级特征。

必须字段：

| 字段 | 说明 |
| --- | --- |
| `position_feature_id` | 明细 ID |
| `measurement_id` | 所属 measurement |
| `hand_side` | 左/右手 |
| `pulse_position` | 寸/关/尺 |
| `feature_name` | 标准变量名 |
| `feature_value` | 数值或文本值 |
| `feature_unit` | 单位 |
| `source_field` | 来源字段名 |
| `parser_version` | 解析版本 |
| `quality_weight` | 质量权重，可空 |

### 3.5 变量字典

已新增 `dim_feature_variable`，让下游不依赖硬编码字段名。

必须字段：

| 字段 | 说明 |
| --- | --- |
| `feature_name` | 机器可读变量名 |
| `display_name` | 中文名 |
| `modality_type` | 模态 |
| `feature_level` | visit/measurement/window/beat/position |
| `source_vendor` | 来源，或 `standard` |
| `data_type` | numeric/categorical/text/json |
| `unit` | 单位 |
| `category` | 时域/频域/形态/质量/中医标签等 |
| `is_ml_feature` | 是否可进入机器学习样本 |
| `is_quality_feature` | 是否为质量控制字段 |
| `valid_range_json` | 合理范围 |
| `description` | 说明 |

## 4. 下游反推的平台必备 API

### 4.1 数据集构建 API

用于将平台数据固化为下游分析版本。

```text
POST /api/datasets
POST /api/datasets/{dataset_id}/versions
GET  /api/datasets/{dataset_id}/versions/{version_id}
GET  /api/datasets/{dataset_id}/versions/{version_id}/manifest
GET  /api/datasets/{dataset_id}/versions/{version_id}/artifact
```

脉诊第一版数据集构建必须支持以下过滤：

| 过滤项 | 说明 |
| --- | --- |
| `modality=pulse` | 仅脉诊 |
| `source_vendor` | 中科/玉生堂 |
| `date_range` | 日期范围 |
| `user_ids` | 用户集合 |
| `visit_slots` | 早/中/晚 |
| `quality_status` | visit 质量 |
| `has_waveform` | 是否有原始波形 |
| `has_position_features` | 是否有六部位明细 |
| `min_duration_seconds` | 最短采集时长 |
| `device_ids` | 设备过滤 |

### 4.2 脉诊 analysis-ready 查询 API

展示 API 不等于分析 API。下游模块需要稳定、扁平、可分页、可导出的接口。

```text
GET /api/pulse/measurements
GET /api/pulse/measurements/{measurement_id}
GET /api/pulse/measurements/{measurement_id}/waveforms
GET /api/pulse/measurements/{measurement_id}/position-features
GET /api/pulse/features
GET /api/pulse/feature-variables
```

`GET /api/pulse/measurements` 至少返回：

| 字段组 | 内容 |
| --- | --- |
| 样本键 | `measurement_id`、`visit_id`、`user_id` |
| 时间 | `start_time`、`duration_seconds`、`visit_slot`、`collection_hour` |
| 设备 | `device_id`、`device_model`、`source_vendor` |
| 采集条件 | `hand_side`、`pulse_position`、`posture`、`operator_id` |
| 质量 | `quality_status`、`quality_flags` |
| 特征摘要 | `feature_json` 或关联 feature 表 |
| 资产 | `waveform_asset_count`、`has_preview`、`has_raw_waveform` |

### 4.3 导出 API

脉诊第一版至少需要三类导出：

```text
GET /api/exports/pulse/measurements.csv
GET /api/exports/pulse/features.parquet
GET /api/exports/pulse/waveform-manifest.jsonl
```

推荐最终由数据集版本统一承载：

```text
storage/datasets/DS-PULSE-QUALITY/vYYYY.MM.NNN/
  dataset_card.md
  manifest.jsonl
  measurements.parquet
  measurement_features.parquet
  position_features.parquet
  waveform_manifest.jsonl
  files/
```

## 5. 分析结果回写接口

闭环不止是平台向下游供数。脉诊分析完成后，应把质量结果、可靠片段和特征可靠性等级回写平台，供展示、筛选和后续数据集构建使用。

### 5.1 建议新增表

| 表 | 用途 |
| --- | --- |
| `analysis_run` | 一次分析运行，记录代码版本、参数、输入数据集版本 |
| `fact_pulse_measurement_quality` | 单次测量质量评分、漂移结果、推荐片段 |
| `fact_pulse_window_quality` | 窗口级质量评分和漂移标记 |
| `fact_pulse_beat_feature` | 搏动级检测和特征 |
| `fact_pulse_feature_reliability` | 特征可靠性分级 |
| `fact_pulse_device_consistency` | 设备一致性结果 |

### 5.2 回写 API

```text
POST /api/analysis-runs
POST /api/analysis-runs/{run_id}/pulse/measurement-quality
POST /api/analysis-runs/{run_id}/pulse/window-quality
POST /api/analysis-runs/{run_id}/pulse/feature-reliability
POST /api/analysis-runs/{run_id}/pulse/device-consistency
GET  /api/analysis-runs/{run_id}
```

第一版最关键回写字段：

| 字段 | 说明 |
| --- | --- |
| `drift_severity_index` | 漂移严重度 |
| `stable_segment_ratio` | 稳定片段比例 |
| `best_segment_start_time` / `best_segment_end_time` | 推荐片段 |
| `best_segment_quality_score` | 推荐片段质量 |
| `signal_quality_score` | 单次测量质量 |
| `measurement_validity_label` | 有效/局部有效/无效 |
| `device_bias` / `device_icc` | 设备一致性 |
| `within_session_cv` / `within_session_icc` | 单次内稳定性 |
| `feature_reliability_grade` | A/B/C/D 可靠性等级 |

## 6. MVP 优先级

为了让脉诊模块尽快反推并验证平台接口，建议把 M2 剩余工作拆成以下顺序：

1. 固化 `fact_pulse_measurement`，让每条脉诊 records 有稳定 `measurement_id`。
2. 固化 `fact_pulse_waveform_asset`，把完整波形从 JSON records 中迁移为文件资产索引。
3. 固化 `fact_pulse_position_feature`，把中科六部位明细从 records 中迁移为高维明细表。
4. 新增 `dim_feature_variable`，统一变量字典和可训练字段标记。
5. 增加脉诊 analysis-ready 导出：`measurements + features + position_features + waveform_manifest`。
6. 增加最小 `dataset_version.manifest_uri/artifact_uri`，让下游按版本消费。
7. 增加分析结果回写表，先支持 measurement 级质量结果和 feature 级可靠性等级。

完成以上 7 项后，数据平台与脉诊分析模块形成第一版闭环：平台可产出数据集，下游可复现实验，下游结果可回流平台并影响后续质量过滤和数据集版本。

当前完成第 1 至第 4 项的数据结构和 demo seed，完成第 5 项的查询接口雏形；CSV/Parquet 文件导出、正式 dataset manifest 和结果回写写入 API 尚未完成。
