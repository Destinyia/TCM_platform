# 多模态数据平台数据结构设计

生成时间：2026-04-22

## 1. 目标

平台目标不是只清洗当前四诊仪数据，而是建设一个可持续扩展的多模态研究数据库，支撑中西医结合研究，并为后续接入更多设备、检验、影像、问卷、可穿戴和随访数据预留统一模型。

当前已经明确的约束：

1. 研究主线是用户纵向随访。
2. 一次 `visit` 下可以同时有多模态数据。
3. 原始文件路径和来源字段必须长期保留，可追溯。
4. 缺失、异常、疑似作弊、重复、姓名映射冲突都要显式建模。
5. 不能把当前中科/玉生堂目录结构硬编码成最终平台模型。

## 2. 入库标准规则

### 2.1 身份与主键规则

用户级主键分三层：

1. 平台内部主键：`user_id`
2. 来源侧用户键：`source_user_key`
3. 来源侧就诊键：`source_visit_id`

当前两套四诊仪规则：

- 中科用户键：`标准姓名 + 手机号`
- 中科 visit 键：`病例号`
- 玉生堂用户键：`标准姓名 + 手机号`
- 玉生堂 visit 键：`TreatNumber`
- 玉生堂辅助来源 ID：`CaId`

说明：

- `CaId` 保留，但不直接作为平台用户主键。
- `病例号` 是 visit 级键，不是用户级键。
- 平台内部 `user_id` 必须独立生成，不直接等于任一来源字段。
- 玉生堂 `TreatNumber`/报告 `档案编号` 在质量核查和入库审计中必须作为第二关键列展示，便于从任意导出表回溯到原始病例目录。
- 当玉生堂结构化 JSON 缺失但存在 `{TreatNumber}.pdf` 报告时，PDF 可作为补充解析来源：提取 `姓名`、`档案编号`、`评估日期`、脱敏电话等字段，写入来源证据字段，并标记 `pdf_identity_fallback`。
- 身份映射优先级：结构化 `CaId` 映射 > PDF 报告姓名 > PDF 脱敏电话；PDF 解析不能覆盖原始 JSON 字段，只能补充空缺字段。

### 2.2 姓名标准化规则

入库前先统一 `raw_name -> canonical_name`。

当前确认的中科别名映射：

- `aln` -> `爱丽娜`
- `爱丽娜·哈力斯` -> `爱丽娜`
- `艾琳娜` -> `爱丽娜`
- `艾琳娜aln` -> `爱丽娜`
- `zyn` -> `张雨楠`
- `ZYN` -> `张雨楠`
- `lyl` -> `李昱霖`
- `蒋` -> `蒋广祥`

当前确认的格式归一化：

- `渠瀚云\n渠瀚云` -> `渠瀚云`
- 去除换行、首尾空格、连续空格
- 保留原始姓名到 `raw_name`
- 归一化后写入 `canonical_name`

要求：

1. 所有别名规则必须版本化管理。
2. 别名规则进入 `dim_user_identity_map` 或独立规则表，不能只写在脚本里。
3. 原始字段绝不覆盖，只增加标准化字段。

### 2.3 数据质量规则

质量标签作为标准字段入库，不作为临时备注。

建议统一标签：

- `missing_modality`
- `parse_failed`
- `duplicate_export`
- `duplicate_numeric_exact`
- `duplicate_numeric_similar`
- `triplicate_within_10m`
- `gap_lt_threshold`
- `name_alias_mapped`
- `name_mismatch`
- `pdf_identity_fallback`
- `report_only_record`

建议统一状态：

- `valid`
- `incomplete`
- `suspicious`
- `invalid`
- `duplicate`

导出与审计规则：

1. 详情表保留原始记录粒度，一个原始病例目录一行。
2. 研究矩阵可以按规则合并为早/中/晚槽位，但不能删除原始行。
3. 同一 `source_vendor + source_visit_id` 多次导出的副本需要保留审计记录，排在详情表末尾，`visit_slot=重复`，质量标签包含 `duplicate_export`。
4. 只有 PDF/图片、缺少结构化 JSON 的记录应入库为 `incomplete`，但保留 PDF 和图片资产，不应直接丢弃。
5. 当前打卡矩阵统计视图按固定时间窗汇总：`早 < 11:00`，`中 11:00-15:59:59`，`晚 >= 16:00`；矩阵单元格显示原始记录次数，而不是只显示 1。

## 3. 分层架构

建议使用四层：

1. `raw`
2. `standard`
3. `feature`
4. `mart`

### 3.1 raw

保留原始文件和原始元数据：

- Excel
- CSV
- JSON
- 图片
- wav
- PDF
- 原始目录结构

### 3.2 standard

平台核心层，统一为以下实体：

- `user`
- `visit`
- `modality_record`
- `file_asset`

### 3.3 feature

保存统一特征：

- 脉诊统计特征
- 舌图特征
- 面诊特征
- 问诊量表特征
- 语音声学特征
- 跨模态融合特征

### 3.4 mart

面向研究和可视化：

- 用户纵向面板
- cohort 统计表
- 依从性表
- 异常表
- 训练数据快照

## 5. 当前两套四诊仪到标准模型的映射

### 5.1 中科

- 问诊表 -> `modality_type=ask`
- 脉诊表/脉波文件 -> `modality_type=pulse`
- 舌诊表/舌图 -> `modality_type=tongue`
- 面诊表/面图 -> `modality_type=face`
- PDF 报告 -> `modality_type=report`

中科关键字段：

- 用户键：`canonical_name + phone`
- visit 键：`病例号`
- 时间字段：`采集时间`

### 5.2 玉生堂

- `dataAsk.json` -> `modality_type=ask`
- `pulse/dataPulse.json` -> `modality_type=pulse`
- `tongue/dataTongue.json` -> `modality_type=tongue`
- `voice/dataVoice.json` -> `modality_type=voice`
- `western/dataWestern.json` -> `modality_type=western_exam`
- PDF -> `modality_type=report`

玉生堂关键字段：

- 用户键：`canonical_name + phone`
- visit 键：`TreatNumber`
- 辅助来源 ID：`CaId`
- 时间字段优先级：记录内部时间 > 目录名时间

## 6. 首版数据库建议

第一阶段建议最少落这 6 张表：

1. `dim_user`
2. `dim_user_identity_map`
3. `fact_visit`
4. `fact_modality_record`
5. `fact_file_asset`
6. `mart_user_day_panel`

这已经足够支撑：

- 纵向依从性分析
- 双设备同日同槽位合并
- 多模态缺失统计
- 异常与作弊标记
- 后续特征工程

## 7. 开发顺序建议

按下面顺序推进最稳：

1. 先建数据库基础表
2. 固化姓名标准化和 ID 生成规则
3. 写 raw -> standard ETL
4. 写质量规则计算
5. 写 `mart_user_day_panel`
6. 再做 API 和平台界面

## 8. 当前明确结论

在正式开发数据库前，以下规则已经可以定稿：

1. 两套四诊仪的用户键都采用 `标准姓名 + 手机号`
2. 中科 `病例号` 作为 visit 键
3. 玉生堂 `TreatNumber` 作为 visit 键
4. `CaId` 只作为辅助来源 ID 保留
5. 中科别名映射必须进入标准入库流程
6. 原始姓名、原始路径、原始来源字段必须完整保留
