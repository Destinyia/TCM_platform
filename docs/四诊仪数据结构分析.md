# 四诊仪数据结构分析

分析对象：`E:\workspace\TCM_platform\四诊仪数据汇总`

生成时间：2026-04-16

## 1. 总体结论

当前数据不是单一导出规范，而是至少 4 种目录组织方式混合：

1. **中科一代“单病例文件夹直放”**：每个病例一个目录，目录内直接放 Excel 汇总表、左右手脉搏原始文件、舌/面/穴位/调理类图片。
2. **中科一代“按患者/批次导出”**：先按患者姓名分目录，再按报告、脉诊、舌诊、面诊、问诊等导出目录组织文件。
3. **中科二代“按模态汇总导出”**：先按模态目录拆分报告、脉诊、舌诊、问诊、面诊，再在模态目录内按日期批次存表格和资产文件。
4. **玉生堂“按检查流水号病例包”**：每个检查流水号一个病例目录，目录内有根级 JSON、模态子目录、PDF、zip/音频/图片；另有根级汇总 Excel。

清洗时不能按扩展名直接选择解析器：中科大量 `.xls` 实际是 OOXML zip 工作簿，问诊 `.xls` 又是真正的 OLE/BIFF Excel；CSV/JSON 需要显式 UTF-8/GB18030 编码探测；长整型 ID 必须按字符串处理，避免被 Excel/Python 读成科学计数法或浮点数。

## 3. 目录结构类型

### 3.1 中科一代：单病例目录直放

典型路径：

```text
中科四诊仪/
  001_2024年12月20日_11时45分/
    001_2024年12月20日_11时45分.xls
    left
    right
    m_<hash>.jpg
    s_<hash>.jpg
    nei_guan.jpg
    qie_men.jpg
    *.png
```

结构特征：

- 顶层下有 640 个病例目录，文件深度固定为 `中科四诊仪/{病例目录}/{文件}`。
- 632 个病例目录同时存在 `left` 和 `right` 无扩展名文件，大小约 493-494 KB，推断为左右手脉搏原始采集数据。
- 每个病例通常有 1 个 `.xls`，文件头为 `PK`，实际是 OOXML 工作簿而不是老式 Excel。
- 工作簿只有 `病例详情` sheet，常见规模为 27 行 x 92 列或 36 行 x 92 列。
- 表内字段覆盖：病例 ID、生成时间、性别、年龄、身高、体重、血压、血糖、出生日期、体质类型/分值、分析结论、饮食调养、运动调理、二十八脉数据等。
- 图片包括：
  - `m_*.jpg`、`s_*.jpg`：疑似面/舌原始或处理图，需结合图像尺寸和人工规则再确认。
  - `nei_guan.jpg`、`qie_men.jpg`、`guan_yuan.jpg`、`qi_hai.jpg` 等：穴位或调理建议图片。
  - 多个 `.png`：命名如 `zqnmzlq.png`、`lspzgsy.png`、`tlpwxdj.png`，更像调理/体质建议素材，不应直接混入诊断原始图。

清洗要点：

- `病例ID` 优先作为 visit/case 主键；目录名里的序号和时间只能作为辅助。
- 原始脉搏文件无扩展名，应按二进制资产入库，后续单独识别格式。
- 图片需要先按命名规则分为“诊断图像”和“建议/素材图像”，否则会污染舌面图像数据集。

结构特征：

- 顶层有 20 个患者姓名目录。
- 子目录命名不完全规范，既有 `冉宇脉诊`、`冉宇舌诊`，也有 `脉2`、`2`、`2025.12.29病历数据(脉诊)` 等变体。
- 数据文件可能同时有 `.xls`、`.xlsx`、`.csv`。部分患者还有一个合并汇总 `.xlsx`，例如面/舌/脉数据合在 105 列宽表中。
- 报告 PDF 为 `1.0报告`。

主要表格类型：

| 类型 | 典型 sheet/文件 | 主要字段 |
| --- | --- | --- |
| 脉诊 | `脉诊数据`，约 49 列 | 序号、报告类型、姓名、性别、年龄、手机号、采集时间、病例号、机构、医生、整体脉象结果、部位、脉位、脉率、节律、脉力、紧张度、流利度、类型、脉位值、脉率值、h1-h5、t/t1/t4/t5、W、As、Ad 等 |
| 舌诊 | `舌诊数据`，约 49 列 | 整体舌象结果、舌色、苔色、胖瘦、厚薄、腐腻、老嫩、点刺、齿痕、裂纹、舌色指数、苔色指数、胖瘦指数、Lab/RGB 类颜色数据等 |
| 面诊 | `面诊数据`，约 36 列 | 整体面诊结果、面色、光泽、唇色、上庭、中庭、下庭、色彩空间、测量标准、全面、鼻、唇、左右眼周、左右眉、左右脸颊、额、下巴等 |
| 问诊 | 真 `.xls` OLE/BIFF | 题库 sheet、体质辨识/简版、五态人格、问诊汇总；问诊汇总含主体质、倾向质、九种体质分值、五态结果等 |

资产类型：

- 脉诊：无扩展名 `left/right` 原始采集文件。
- 舌诊：`原图`、`分割`、`分区` JPG。
- 面诊：`原图`、`分区` JPG；文件名时间常为英文 Java Date 字符串。
- 报告：PDF。

清洗要点：

- 目录名只能辅助判别模态，不能严格依赖；应优先从文件名/sheet 名识别。
- `.csv` 与 `.xls/.xlsx` 可能是同一数据重复导出，需要以 `病例号 + 采集时间 + 模态 + 部位/资产类型` 去重。
- CSV 默认读取可能出现舌诊表头乱码，应显式按 UTF-8/GB18030 尝试并用标准表头校验。

## 5. 关键清洗风险

1. **扩展名不可信**：`.xls` 既可能是 OOXML zip，也可能是 BIFF/OLE。必须读文件头选择解析器。
2. **编码不统一**：CSV 在系统默认编码下会出现乱码；JSON 应优先 UTF-8/UTF-8-SIG。建议按 `utf-8-sig -> utf-8 -> gb18030` 顺序尝试，并用字段字典校验。
3. **ID 精度风险**：病例号、评估编号、TreatNumber 都是长整型，Excel/CSV 读取时必须作为字符串，否则可能变成 `1.437...e+18`。
4. **同一病例重复导出**：中科二代每日和跨日文件有重叠，中科患者目录也可能同时有 CSV、XLS、合并 XLSX。应以源系统病例号/检查号去重。
5. **资产与诊断混杂**：中科一代目录里的穴位/调理图片和诊断图像混放，不能把所有 JPG/PNG 都当作舌面诊图片。
6. **病例完整度不齐**：玉生堂大量病例缺 voice/western/numberOrder 或 PDF；中科部分目录缺 left/right。标准化表要允许模态缺失。
7. **嵌套 JSON 字符串**：玉生堂 `AnalysisResults`、`Detail` 等字段是字符串形式的 JSON，需要二次解析并容错。
8. **PII 合规风险**：姓名、手机号、证件号、生日、地址等广泛存在。清洗产物应拆分 PII 表，并对分析集使用脱敏 ID。

## 6. 建议标准化落地结构

建议先建立“宽松标准层”，保留源路径、源字段和解析状态，避免一次性强行规范导致数据丢失：

核心字段建议：

- `source_vendor`：`zhongke` / `yushengtang`
- `source_batch`：顶层目录名
- `source_path`：原始相对路径
- `source_file_type`：`xlsx_zip_xls` / `biff_xls` / `csv` / `json` / `pdf` / `image` / `raw_binary` / `zip` / `wav`
- `visit_id_source`：中科 `病例ID/病例号`，玉生堂 `TreatNumber/评估编号`
- `patient_id_source`：中科可由病例表中姓名/手机号/证件号组合映射，玉生堂使用 `CaId`
- `collected_at`：采集时间/评估时间/病例生成时间
- `modality`：`ask` / `pulse` / `tongue` / `face` / `voice` / `report` / `prescription` / `western` / `basic`
- `side`：脉诊左右手或空
- `asset_type`：`pulse_raw_left`、`pulse_raw_right`、`tongue_origin`、`tongue_segmented`、`tongue_region`、`face_origin`、`face_region`、`voice_zip`、`voice_wav`、`report_pdf` 等
- `parse_status`：`ok` / `partial` / `failed`
- `raw_payload` 或 `raw_json`：保留无法完全结构化的原始字段
