# 中医多模态数据平台开发计划

更新时间：2026-05-20

## 1. 当前约定

本仓库以当前实现为准，后续开发必须遵守以下约定：

1. 每次修改代码、接口或数据结构后，同步更新相关文档。
2. 文档更新与代码修改必须在同一个 commit 中提交。
3. 涉及前端路由、API 路由、数据库表、接口响应字段、展示聚合规则时，优先更新 `docs/api.md`、`docs/多模态数据展示平台总体设计.md` 和本文。
4. 提交前确认只纳入本次相关文件，不提交本地 dump、临时文件、原始数据和环境目录。

## 2. 当前技术栈

| 层级 | 当前实现 | 说明 |
| --- | --- | --- |
| 后端 | Flask | `backend.app.create_app()` 注册 API 蓝图 |
| API 前缀 | `/api/demo`、`/api/ingest`、`/api/health` | 一阶段 demo 以 `/api/demo` 为主 |
| 数据库 | PostgreSQL + SQLAlchemy | 不采用 MongoDB |
| 文件资产 | 本地 `storage/` 根目录 | 生产环境可配置到 Ubuntu 本地盘或对象存储 |
| 前端 | Vue 3 + Element Plus + ECharts + Vite | 当前代码在 `frontend/src` |
| 开发环境 | Windows 仓库 + WSL 运行前端/生产近似环境 | 生产目标为 Ubuntu |

## 3. 当前核心数据结构

当前 ORM 定义位于 `backend/app/models.py`。

| 表 | 模型 | 用途 |
| --- | --- | --- |
| `dim_user` | `User` | 标准用户/患者主表，患者详情路由使用 `user_id` |
| `dim_user_identity_map` | `UserIdentityMap` | 来源平台用户身份映射 |
| `dim_name_alias_rule` | `NameAliasRule` | 姓名别名和规则版本 |
| `fact_visit` | `Visit` | 一次标准 visit，包含来源、日期、早中晚时段、质量状态 |
| `fact_modality_record` | `ModalityRecord` | visit 下的问诊、脉诊、舌诊、面诊、声诊、报告等模态记录 |
| `fact_file_asset` | `FileAsset` | 图片、PDF、音频、导出物等文件资产元数据 |
| `fact_quality_event` | `QualityEvent` | 质量事件、异常、规则证据 |
| `mart_user_day_panel` | `UserDayPanel` | 用户-日期-时段展示/分析宽表 |
| `dataset_version` | `DatasetVersion` | 数据集版本 demo 元数据 |

关键字段约定：

- `User.user_id` 是前端患者详情页主键。
- `Visit.visit_slot` 取值为 `早`、`中`、`晚`。
- `Visit.quality_status` 当前常见值为 `valid`、`incomplete`、`suspicious`。
- `ModalityRecord.modality_type` 当前包括 `ask`、`pulse`、`tongue`、`face`、`voice`、`report`。
- 文件本体不写入数据库，数据库只保存 `storage_uri`、路径、hash、大小、mime type 等元数据。

## 5. 打卡矩阵当前规则

当前实现位于 `frontend/src/views/CheckinMatrix.vue`，后端接口为 `GET /api/demo/checkin-matrix`。

后端仍返回用户、日期、早中晚时段三层结构：

```text
rows[].cells[visit_date][visit_slot] = {
  count,
  valid_count,
  sources,
  quality_statuses,
  modalities,
  visit_ids,
  visit_times,
  status
}
```

前端展示规则：

1. 展示维度为 `用户 × 日期`，不再把早中晚拆成三个展示列。
2. 同一用户同一天的 `早`、`中`、`晚` 单元格在前端聚合。
3. 单元格只展示 `数字`、`有效/无效`、`来源`。
4. 聚合后 `count = 当天早中晚记录数之和`。
5. 聚合后 `valid_count = 当天早中晚有效记录数之和`。
6. 当 `count == valid_count` 时显示 `有效`，否则显示 `无效`。
7. 无打卡的空单元格留空，不显示 `0`、`无效` 或来源占位。
8. 缩略图和展开视图使用相同网格布局，只调整单元格密度。
9. 姓名区域可点击，跳转到 `/patients/{user_id}`。

## 6. 一阶段目标

一阶段不做自动化管线，采用本地已有静态 mock 数据构造 PostgreSQL demo 数据库，前端通过 Flask API 动态读取。

当前重点：

1. 完善多模态数据呈现形式。
2. 固化用户、visit、模态、资产、质量、数据集版本的最小结构。
3. 打通患者纵向随访、visit 多模态详情、打卡矩阵、脉诊分析的核心接口。
4. 用脉诊模块作为独立下游消费者样例，验证平台是否能产出通用、版本化、可下载的数据集。
5. 形成后续正式 ETL 可复用的数据结构和接口形态。

## 7. 本地运行流程

后端和数据库：

```bash
docker compose up -d postgres
pip install -r requirements.txt
python -m backend.scripts.init_db
python -m backend.scripts.seed_demo_data
python -m backend.run
```

前端建议在有 Node.js 的 WSL 环境运行：

```bash
cd /mnt/e/workspace/TCM_platform/frontend
npm install
VITE_API_BASE=http://localhost:5000 npm run dev -- --host 0.0.0.0
```

生产环境目标为 Ubuntu。后端也建议在 WSL/Ubuntu 语义环境下验证，减少 Windows 与 Linux 在路径、权限、依赖上的差异。
